构建你自己的代码解释器 - 使用 o3-mini 进行动态工具生成和执行

提供 LLM Agent 与外部世界或其他 Agent 交互能力的核心是“工具（或函数）调用”，即 LLM 可以调用一个（代码块）函数并传入参数。通常，这些函数由开发者预先定义，包括其预期的输入和输出。然而，在本指南中，我们探索一种更灵活的范式——使用 LLM 模型（在本例中为 o3-mini）动态生成工具，并能够使用代码解释器执行该工具。

使用代码解释器进行动态生成的工具调用

动态生成的工具是 LLM 在运行时根据用户提示自行创建的函数或代码块。这意味着您不必在代码库中预先定义每一种可能的情况——从而实现更开放、更具创意和适应性的问题解决。

动态生成的工具调用更进一步，它允许 LLM 动态生成工具并即时执行代码块。这种动态方法对于涉及以下任务特别有用：

  • 数据分析和可视化
  • 数据操作和转换
  • 机器学习工作流的生成和执行
  • 流程自动化和脚本编写
  • 以及更多，随着实验的进行，新的可能性不断涌现

使用 o3-mini 进行动态工具生成

o3-mini 模型于 2025 年 1 月 31 日发布，在 STEM 领域拥有卓越的能力——尤其在科学、数学和编码方面表现出色——同时保持了小型模型的低成本和低延迟。在本指南中，我们将演示 o3-mini 生成 Python 代码以解释数据并提取见解的能力。

推理模型在生成动态工具以分析数据方面特别擅长，因为它们可以自行推理，而无需明确的思维链提示。事实上，提供明确的思维链指令可能会干扰模型的内部推理，并导致次优结果。您可以在此处了解更多关于 o3-mini 的信息。

为什么构建自己的代码解释器

许多 API 提供商（例如 OpenAI 的 Assistants API）都提供内置的代码解释器功能。这些内置的代码解释器功能强大无比，但在某些情况下，开发者可能需要创建自己的自定义代码解释器。例如：

  1. 语言或库支持：内置解释器可能不支持您的任务所需的特定编程语言（例如 C++、Java 等）或库。
  2. 任务兼容性：您的用例可能与提供商的内置解决方案不兼容。
  3. 模型限制：您可能需要提供商的解释器不支持的语言模型。
  4. 成本考虑：代码执行或模型使用的成本结构可能不符合您的预算或限制。
  5. 文件大小：输入数据的文件大小过大或提供商的解释器不支持。
  6. 与内部系统集成：提供商的解释器可能无法与您的内部系统集成。

您将学到什么

通过遵循本指南，您将学会如何：

  • 使用 Docker 设置隔离的 Python 代码执行环境
  • 为 LLM Agent 配置您自己的代码解释器工具
  • 建立清晰的“Agentic”关注点分离，以确保安全
  • 使用 o3-mini 模型动态生成数据分析代码
  • 协调 Agent 以高效完成给定任务
  • 设计一个能够动态生成和执行代码的 Agentic 应用程序

您将学会如何从头开始构建自定义代码解释器工具，利用 LLM 的强大功能生成复杂的代码，并在隔离的环境中安全地执行该代码——所有这些都是为了使您的 AI 驱动的应用程序更灵活、更强大、更具成本效益。

示例场景

我们将使用在交通事故关键因素提供的示例数据来回答一系列问题。这些问题不需要预先定义，我们将赋予 LLM 生成代码来回答这些问题的能力。

示例文本可能包括：

  • 哪些因素对事故频率的贡献最大？（特征重要性分析）
  • 哪些区域的事故风险最高？（分类/聚类）
  • 交通罚款金额如何影响事故数量？（回归/因果推断）
  • 能否确定最佳罚款金额以降低事故率？（优化模型）
  • 更高的罚款是否与较低的平均速度或较少的事故相关？（相关性/回归）
  • 等等……

使用传统的预定义工具调用方法，开发者需要为每个问题预定义函数。这限制了 LLM 回答预定义函数集之外的任何其他问题的能力。我们通过使用动态工具调用方法来克服这一限制，在该方法中，LLM 生成代码并使用代码解释器工具执行代码。

概述

让我们深入了解构建此 Agentic 应用程序（具有动态工具调用）的步骤。该应用程序包含三个组件：

步骤 1：设置隔离的代码执行容器环境

我们需要一个安全的执行 LLM 生成的函数调用的环境。我们希望避免直接在主机上运行 LLM 生成的代码，因此我们将创建一个资源访问受限（例如，无网络访问）的 Docker 容器环境。默认情况下，Docker 容器无法访问主机的文件系统，这有助于确保 LLM 生成的代码保持在容器内。

⚠️ 警告：为 LLM 生成的代码实施强大的防护措施

LLM 可能会生成具有意外后果的有害代码。作为最佳实践，请隔离代码执行环境，仅授予任务所需的最低资源访问权限。避免在您的主机或笔记本电脑上运行 LLM 生成的代码。

步骤 2：定义和测试 Agent

“什么是 Agent？” 在本指南的上下文中，Agent 是：

  1. LLM 遵循的指令集，即开发者提示
  2. 一个 LLM 模型，以及通过 API 调用模型的
  3. 工具调用函数的能力，以及执行函数的能力

我们将定义两个 Agent：

  1. FileAccessAgent：此 Agent 将读取文件并将内容作为上下文提供给 PythonCodeExecAgent。
  2. PythonCodeExecAgent：此 Agent 将生成 Python 代码以回答用户的问题，并在 Docker 容器中执行该代码。

步骤 3：设置 Agentic 编排以运行应用程序

可以根据应用程序需求以各种方式编排 Agent。在此示例中，我们将使用简单的编排，其中用户提供任务，然后按顺序调用 Agent 来完成任务。

  1. Docker：已安装并正在运行。您可以在此处了解更多关于 Docker 的信息并进行安装。
  2. Python：已安装在您的本地计算机上。您可以在此处了解更多关于 Python 的信息并进行安装。
  3. OpenAI API 密钥：已设置为本地计算机上的环境变量或根目录中的 .env 文件。您可以在此处了解更多关于 OpenAI API 密钥的信息并进行设置。

步骤 1：设置隔离的代码执行环境

让我们定义一个将用于执行代码的 Docker 化容器环境。我在 resources/docker 目录下的 Dockerfile 中定义了一个用于创建容器环境的 Dockerfile，其规格如下：

  • 基于 Python 3.10
  • 一个非 root 用户
  • 预先安装 requirements.txt 中的包

在 Docker 镜像创建过程中包含的 requirements.txt 中包含了我们 LLM 生成的代码可能需要的所有潜在包。鉴于我们将限制容器的网络访问，因此我们需要预先安装任务所需的包。出于安全原因，我们的 LLM 将不允许安装任何其他包。

您可以创建自己的 Docker 镜像，其中包含语言要求（如 Python 3.10）并预先安装任务所需的包，或者创建自定义 Docker 镜像，其中包含特定语言（如 Java、C++ 等）和任务所需的包。

步骤 2：定义和测试 Agent

为了我们的目的，我们将定义两个 Agent。

  1. Agent 1：文件访问 Agent（带预定义工具调用） - 指令是理解文件内容，将其作为上下文提供给 Agent 2。 - 可以访问主机的文件系统。 - 可以从主机读取文件并将其复制到 Docker 容器中。 - 无法访问代码解释器工具。 - 使用 gpt-4o 模型。

  2. Agent 2：Python 代码生成器和执行器（带动态生成的工具调用和代码执行） - 从 Agent 1 接收文件内容上下文。 - 指令是生成一个 Python 脚本来回答用户的问题。 - 可以访问 Docker 容器内的代码解释器，用于执行 Python 代码。 - 只能访问 Docker 容器内的文件系统（不能访问主机）。 - 无法访问主机的文件系统或网络。 - 使用我们最新的 o3-mini 模型，该模型在代码生成方面表现出色。

这种文件访问（Agent 1）和代码生成器和执行器（Agent 2）之间的关注点分离对于防止 LLM 直接访问或修改主机至关重要。

将 Agent 1 限制为静态工具调用，因为它具有对主机文件系统的访问权限。

| Agent | 工具调用类型 | 主机文件系统访问权限 | Docker 容器文件系统访问权限 | 代码解释器访问权限 |

Agent 工具调用类型 主机文件系统访问权限 Docker 容器文件系统访问权限 代码解释器访问权限
Agent 1: 文件访问 预定义工具 是 是 否
Agent 2: Python 代码生成器和执行器 动态生成的工具 否 是 是

为了保持 Agent 和工具的组织性，我们定义了一组核心类，用于创建这两个 Agent，以保持一致性并使用面向对象编程原则。

  • BaseAgent：我们从一个抽象基类开始，它强制执行通用的方法签名，例如 task()。基类还提供用于调试的日志记录器、语言模型接口以及其他通用函数，例如用于向 Agent 添加上下文的 add_context()。
  • ChatMessages：一个用于存储对话历史的类，因为 ChatCompletions API 是无状态的。
  • ToolManager：一个用于管理 Agent 可以调用的工具的类。
  • ToolInterface：任何 Agent 可以调用的“工具”的抽象类，以便工具具有一致的接口。

这些类定义在 resources/object_oriented_agents/core_classes 目录下的 object_oriented_agents/core_classes 中。

定义 Agent 1：FileAccessAgent 和 FileAccessTool

让我们开始定义继承自 ToolInterface 类的 FileAccessTool。FileAccessTool 工具定义在 resources/registry/tools 目录下的 file_access_tool.py 文件中。

  • FileAccessTool 实现 ToolInterface 类，这确保了工具将具有一致的接口。
  • 为 OpenAI 函数调用 API 绑定工具定义，在 get_definition 方法和工具的 run 方法中，确保了可维护性、可扩展性和可重用性。

现在，让我们定义扩展 BaseAgent 类的 FileAccessAgent，并将 FileAccessTool 绑定到该 Agent。FileAccessAgent 定义在 resources/registry/agents 目录下的 file_acess_agent.py 文件中。FileAccessAgent 是：

  • BaseAgent 类的具体实现。
  • 使用开发者提示、模型名称、日志记录器和语言模型接口进行初始化。开发者可以根据需要覆盖这些默认值。
  • 具有 setup_tools 方法，该方法将 FileAccessTool 注册到工具管理器。
  • 具有 task 方法，该方法调用 FileAccessTool 来读取文件并将上下文提供给 PythonCodeExecAgent。
  • model_name='gpt-4o'，它提供了足够的推理和工具调用能力来完成任务。

定义 Agent 2：PythonExecAgent 和 PythonExecTool

类似地，PythonExecTool 继承自 ToolInterface 类，并实现 get_definition 和 run 方法。get_definition 方法以 OpenAI 函数调用 API 期望的格式返回工具定义。run 方法在 Docker 容器中执行 Python 代码并返回输出。此工具定义在 resources/registry/tools 目录下的 python_code_interpreter_tool.py 文件中。

同样，PythonExecAgent 是 BaseAgent 类的具体实现。它定义在 resources/registry/agents 目录下的 python_code_exec_agent.py 文件中。PythonExecAgent 是：

  • BaseAgent 类的具体实现。
  • 使用开发者提示、模型名称、日志记录器和语言模型接口进行初始化。开发者可以根据需要覆盖这些默认值。
  • 具有 setup_tools 方法，该方法将 PythonExecTool 注册到工具管理器。
  • 具有 task 方法，该方法调用 OpenAI API 来执行用户的任务，在本例中涉及生成 Python 脚本来回答用户的问题并使用 Code Interpreter 工具运行它。
  • model_name='o3-mini'，它在 STEM 任务（如代码生成）方面表现出色。
  • reasoning_effort='high'，这允许进行更完整的推理，因为任务的复杂性会增加令牌生成和响应速度的成本。默认值为中等，这是速度和推理准确性之间的平衡。

您可以在此处了解更多关于 reasoning_effort 参数的信息。

步骤 3：设置 Agentic 编排以运行应用程序

在定义了 Agent 之后，我们现在可以定义运行应用程序的编排循环。此循环将提示用户输入问题或任务，然后调用 FileAccessAgent 读取文件并将上下文提供给 PythonExecAgent。PythonExecAgent 将生成 Python 代码来回答用户的问题，并在 Docker 容器中执行该代码。代码执行的输出将显示给用户。

用户可以输入“exit”来停止应用程序。我们的问题：哪些因素对事故频率的贡献最大？ 请注意，我们没有预先定义回答此问题的功能。

# Import the agents from registry/agents

from resources.registry.agents.file_access_agent import FileAccessAgent
from resources.registry.agents.python_code_exec_agent import PythonExecAgent


prompt = """Use the file traffic_accidents.csv for your analysis. The column names are:
Variable    Description
accidents   Number of recorded accidents, as a positive integer.
traffic_fine_amount Traffic fine amount, expressed in thousands of USD.
traffic_density Traffic density index, scale from 0 (low) to 10 (high).
traffic_lights  Proportion of traffic lights in the area (0 to 1).
pavement_quality    Pavement quality, scale from 0 (very poor) to 5 (excellent).
urban_area  Urban area (1) or rural area (0), as an integer.
average_speed   Average speed of vehicles in km/h.
rain_intensity  Rain intensity, scale from 0 (no rain) to 3 (heavy rain).
vehicle_count   Estimated number of vehicles, in thousands, as an integer.
time_of_day Time of day in 24-hour format (0 to 24).
accidents   traffic_fine_amount
"""


print("Setup: ")
print(prompt)

print("Setting up the agents... ")

# Instantiate the agents with the default constructor defined values
# Developer may override the default values - prompt, model, logger, and language model interface if needed

# This agent use gpt-4o by default
file_ingestion_agent = FileAccessAgent()

# Let's make sure agent uses o3-mini model and set the reasoning_effort to high
data_analysis_agent = PythonExecAgent(model_name='o3-mini', reasoning_effort='high')

print("Understanding the contents of the file...")
# Give a task to the file ingestion agent to read the file and provide the context to the data analysis agent 
file_ingestion_agent_output = file_ingestion_agent.task(prompt)

# Add the file content as context to the data analysis agent
# The context is added to the agent's tool manager so that the tool manager can use the context to generate the code 

data_analysis_agent.add_context(prompt)
data_analysis_agent.add_context(file_ingestion_agent_output)

while True:

    print("Type your question related to the data in the file. Type 'exit' to exit.")
    user_input = input("Type your question.")

    if user_input == "exit":
        print("Exiting the application.")
        break

    print(f"User question: {user_input}")

    print("Generating dynamic tools and using code interpreter...")
    data_analysis_agent_output = data_analysis_agent.task(user_input)

    print("Output...")
    print(data_analysis_agent_output)

在此示例中，o3-mini 根据用户的提问动态生成了一个工具（Python 脚本）来分析数据。请注意，o3-mini 使用多种方法（如相关性分析、线性回归和随机森林模型）来检查问题。此方法突出了以下几点：

reasoning_effort：模型执行的推理深度，例如在此案例中涉及的方法数量，通常在参数从低、中增加到高时会增加。您可以尝试不同的推理级别来查看差异。

动态生成的工具调用：用于分析数据的工具（Python 脚本）不是手动编写或由开发人员预先确定的。相反，o3-mini 模型在运行时创建了相关的数据探索和相关性分析代码。

隔离的代码执行：为确保安全并避免在主机上运行不受信任的代码，Python 脚本在 Docker 容器中使用 execute_python_code 工具执行。该容器具有受限的资源访问权限（例如，无网络和有限的文件系统访问权限），从而最大限度地降低了任意代码执行带来的潜在风险。

结论

本指南提供了开发定制化自定义代码解释器以满足特定应用程序需求的指南，解决了供应商提供的解决方案中的限制，例如语言限制、成本考虑以及与不同 LLM 或模型保持灵活性的需求。

管理 Agent 和工具的方法：我们还定义了一组核心类来管理 Agent 和工具。这种方法确保 Agent 和工具将具有一致的接口，并且可以在不同的应用程序中重用。可以创建一个 Agent 和工具存储库，例如注册表文件夹，来管理 Agent 和工具。

o3-mini 模型：我们演示了 o3-mini 模型根据用户最少的提示在运行时生成复杂代码以分析数据的能力。然后，o3-mini 模型对分析结果进行推理，向用户解释结果。

最后，总结一下，构建具有动态工具调用的 Agentic 应用程序的三个步骤是：

  1. 设置隔离的代码执行容器环境
  2. 定义和测试 Agent
  3. 设置 Agentic 编排以运行应用程序
